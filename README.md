# corporate-site

hibou-web.com の構成見直しメモ。

このリポジトリは、Hibou Web のコーポレートサイトおよび LP 再構成の設計メモ、実装タスク、今後の VTDD ログ連携の起点として使う。

---

## 目的

- `hibou-web.com/` のトップページを、現在の `/before-update/` の思想に寄せて再構成する
- 「WordPress 修理屋」ではなく、**更新前確認** を主軸にした第一印象へ切り替える
- 問い合わせ時の心理ハードルを下げるため、実務情報を追加する
- FAQ と JSON-LD を実装し、検索結果での理解促進を狙う
- 将来的に VTDD の流れに載せ、思考・判断・変更履歴をログ化できるようにする

## 現状認識

### 良い点

- `/before-update/` は思想が強い
- 「アップデートは、事前に終わらせる」
- 「更新していいかを、先に答えます」
- 「アップデート代行ではなく判断サービス」という軸がある

### 課題

- 現在のトップページと `/before-update/` の価値提案が分断している
- `/before-update/` は思想 LP としては強いが、発注前の不安解消情報が足りない
- 単発チェックや相談導線の具体像がまだ弱い

## サイト構成変更方針

### 1. トップページを `/before-update/` ベースへ差し替える

トップページで最初に見せる価値は以下。

- 更新していいかを、先に答える
- 更新は、実行ではなく確認から
- アップデートは、事前に終わらせる

### 2. 現在のトップページ内容は別固定ページへ移設する

候補:

- `/wp-update/`
- `/site-update/`
- `/maintenance/`

※ `/wordpress-update/` は商標ポリシー解釈の余地があり、避ける方が安全。

### 3. `/before-update/` に「問い合わせ前の不安を潰す情報」を追加する

追加対象:

- 必要な情報
- 初回の流れ
- 返信までの目安
- 受けられない条件
- 相談だけでもよいか

### 4. FAQ を追加する

FAQ 候補:

- 相談だけでも可能ですか？
- 最初に何を送ればよいですか？
- 返信までの目安はどのくらいですか？
- 受けられないケースはありますか？
- 単発チェックでは何が返ってきますか？
- ステージングとの違いは何ですか？
- バックアップとの違いは何ですか？
- 更新代行との違いは何ですか？

### 5. FAQ 用 JSON-LD を実装する

- `FAQPage` を使用
- ページ本文に実際に表示している FAQ と一致させる
- リッチリザルト表示は保証されない前提で入れる
- 実装後は Rich Results Test で確認する

## `/before-update/` に追加するコンテンツ案

### ご相談前にご確認いただきたいこと

#### 必要な情報

- サイト URL
- ご相談内容
- 更新したい対象
  - WordPress 本体
  - テーマ
  - プラグイン
  - PHP バージョン
- 現在把握している懸念点
- 可能なら管理情報や環境情報（必要に応じて個別案内）

#### 初回の流れ

1. フォームまたは連絡手段から相談
2. 対象サイトとご相談内容を確認
3. 実施可否と進め方を案内
4. 更新前確認を実施
5. 判定結果と次のアクションを共有

#### 返信までの目安

- 通常は内容確認後、できるだけ早く返信
- 混雑時や調査内容によっては時間を要する場合あり
- 目安時間を具体的に出せるなら今後固定化する

#### 相談だけでもよいか

- 相談段階でも可
- 更新前確認が必要かどうか迷っている段階でも受付可

#### 受けられない条件

- 必要な情報をご提供いただけない場合
- 緊急復旧のみを目的とした依頼
- すでに本番で重大障害が発生しており、事前確認ではなく復旧対応が主目的の場合
- 環境的に必要な確認手段が取れない場合

## VTDD 連携の意図

このリポジトリは、単なるサイト修正用ではなく、将来的に VTDD で以下を残す起点にする。

- 何を考えて変更したか
- どのページをどう位置づけたか
- どの判断でトップを差し替えたか
- どの課題に対して FAQ を追加したか
- 実装と判断のログ

理想の流れ:

1. ここで議論する
2. 構成案をリポジトリに記録する
3. 実装タスクへ分解する
4. 変更を加える
5. 差分と意図をあとから追えるようにする
